## 🎯 목표

지금은 프로그램(터틀 봇)이 직접 낸 주문만 체결 원장(`trade_ledger.json` + 구글 시트)에 기록됩니다.
사용자가 **손(HTS·MTS)으로 직접 사고판 종목**은 원장에 안 남아서, 손익 집계가 비게 됩니다.

이 계획은 **프로그램이 실행될 때마다, 실제 잔고와 기록(`held_stock_record.json`)이 안 맞는 종목만 골라서**
LS API로 그 종목의 체결 내역을 받아와 **스프레드시트(체결 원장)에 자동으로 채워 넣는 것**입니다.

핵심 원칙:
- 잔고가 안 맞는 종목만 대상 (전 종목을 매번 긁지 않음 → API 부담·중복 최소화)
- 이미 기록된 체결은 다시 안 넣음 (`record_id` 중복 방지)
- 수동 매수·수동 매도 둘 다 커버

### P1-1 자료 수집 ⬜

> 목적: 작업에 필요한 파일, 코드, API 정보를 빠짐없이 모은다.

#### 확인 항목
- [ ] `balance_sync.py` — 실제 잔고 ↔ `held_stock_record.json` 동기화 로직, "안 맞는 종목" 판별 방식
- [ ] `trade_ledger.py` — `append_trade(record)` 진입점, `record_id` 생성·중복 방지 방식, 시트 동기화
- [ ] `ls_client.py` — 체결 내역(체결 조회) API 제공 여부 / 함수명 / 반환 구조
- [ ] `held_stock_record.json` 형식 — 어떤 필드로 보유 수량·평균가를 들고 있는지
- [ ] `trade_ledger.json` 형식 — 기존 기록 구조, `source` 값 종류
- [ ] `run_all.py` — balance_sync가 언제·어떻게 호출되는지 (실행 시점)
- [ ] `.env` — 계좌·시트 관련 환경변수 확인

#### 수집 결과 ✅

**1) `balance_sync.py` — 완벽한 연결 지점**
- `run_balance_sync()`가 실행 시작 시 실제 잔고와 `held_stock_record.json`을 비교한다.
- 이미 "잔고 안 맞는 종목"을 3가지로 잡아낸다:
  - ① 기록엔 있는데 실제로 없음 → **수동 전량 매도**로 판단, 기록 삭제 (114~123줄)
  - ② 실제로 있는데 기록에 없음 → **수동 매수**로 판단, 자동 편입 (128~177줄)
  - ③ 둘 다 있는데 수량 다름 → 수량 수정 (182~195줄) — 부분 수동 매수/매도
- → 이 세 군데가 "체결 기록 받아오기"를 끼워 넣을 자리다.

**2) `trade_ledger.py` — 기록 진입점은 이미 있음**
- `append_trade(record)` 단일 진입점 + 구글시트 동기화 완비.
- `source` 허용값에 `MANUAL_SYNC` 이미 포함 (43줄).
- ⚠️ 주의: `record_id`가 `시각_종목코드_랜덤4자리`라 **매번 달라짐 → 재실행 시 같은 체결이 중복 기록될 위험**. 체결번호 기반 중복 방지 필요.

**3) `ls_client.py` — 체결내역 조회 API 없음 (신규 추가 필요)**
- 현재 accno 계열은 t0424(잔고)만 래핑됨. 체결내역 조회 함수 없음.
- `programgarden_finance` 패키지에 체결내역 조회 TR 2개 확인:
  - **`t0425` (주식 체결/미체결)**: **당일만**. 행별 expcode·ordno·medosu(1매도/2매수)·qty·cheqty(체결수량)·cheprice(체결가)·ordtime. 체결번호 없음.
  - **`CDPCQ04700` (기간별 체결내역)**: **날짜 범위 조회**. 행별(OutBlock3) TrdDt(체결일)·TrdNo(체결번호)·IsuNo·IsuNm·TrdQty·TrdPrc·BnsTpCode(매수/매도)·CmsnAmt(수수료)·EvrTax(세금)·AdjstAmt(실수령). **종목코드(IsuNo)로 필터 가능.**
- → CDPCQ04700이 (a)전날 거래도 잡고 (b)체결번호로 중복 방지 가능 (c)수수료·세금·실수령까지 줌 → **더 적합**.

**4) 인증 방식**
- 기존 t0424 호출은 계좌번호·비밀번호 없이 동작 (라이브러리가 토큰으로 처리).
- ⚠️ CDPCQ04700 InBlock에 `Pwd`(비밀번호) 필드 존재. `.env`에는 계좌 비밀번호 항목 없음 → **빈 값으로 동작하는지 AWS에서 실측 필요** (로컬엔 파이썬·패키지 미설치).

**5) 실행 환경**
- 이 PC(Windows)엔 파이썬/패키지 미설치 → 실제 실행·테스트는 AWS 서버에서. 로컬에선 코드 작성·정적 검토만 가능.

### P1-2 자료 검증 ✅

> 목적: 모은 자료가 정확한지 확인한다. 잘못된 정보가 섞여 있으면 나중에 코드가 오작동한다.

#### 확인 항목
- [ ] LS 체결 조회 API가 실제로 존재하고 "특정 종목·기간"으로 조회 가능한지 확인
- [ ] API 반환값에 기록에 필요한 필드(체결가·수량·시각·매수/매도 구분·체결번호)가 다 있는지 확인
- [ ] `record_id` 중복 방지가 체결번호 기반으로 가능한지 확인
- [ ] "잔고 안 맞음" 판별이 수동 매수/매도 양쪽을 다 잡아내는지 확인
- [ ] 실계좌/모의투자 설정 확인

#### 검증 결과 ✅

- **연결 지점 검증**: `balance_sync.run_balance_sync()`가 실행 초기에 호출되며(run_all.py), 세 가지 불일치를 이미 분기 처리 → 추가 분기 없이 그 자리에 기록 호출만 끼우면 됨. ✔
- **중복 방지 검증 (가장 중요)**: t0425는 그날의 **체결된 주문**을 행마다 `ordno`(LS 주문번호)와 함께 돌려준다. 봇이 직접 낸 주문도 같은 `ordno`로 원장에 이미 기록돼 있다(`order_no` 필드).
  → **"종목코드 + order_no + 오늘 날짜"가 원장에 이미 있으면 건너뛴다**는 한 가지 규칙으로
  ① 봇 자기 주문 재기록 방지 ② 재실행 시 수동체결 중복 방지 둘 다 해결됨. ✔
- **인증 검증**: t0425는 t0424와 같은 accno 계열 → 계좌 비밀번호 불필요(키만으로). `.env` 변경 없음. ✔
- **수익 계산 검증**: 수동 전량매도(case ①)는 기록 삭제 **직전** `held[code]["avg_buy_price"]`가 남아있음 → 이걸로 수익률·수익금 계산 가능. ✔
- **실계좌 위험**: 이 기능은 **조회(t0425) + 원장 기록**만 한다. 주문 함수 호출 없음 → 실계좌 매매 위험 없음. ✔

#### 데이터 흐름도

```
run_all.py 실행
  └─ balance_sync.run_balance_sync()
       ├─ 실제 잔고(actual) vs 기록(held) 비교
       ├─ 불일치 종목 목록 추림 (①삭제대상 ②신규 ③수량차이)
       │     └─ [신규] 불일치 종목마다 avg_buy_price 미리 확보 (삭제 전)
       ├─ (기존 동기화 로직: 삭제/편입/수량수정 그대로)
       └─ [신규] 불일치 종목마다:
            _record_manual_executions(code, name, avg_buy_price)
              ├─ ls_client.get_today_executions(code)   ← t0425 (당일 체결)
              └─ 체결 행마다:
                   ├─ trade_ledger.order_already_recorded(code, ordno)?
                   │     └─ 있으면 skip (봇 자기주문·재실행 중복 방지)
                   └─ 없으면 trade_ledger.append_trade({
                          side, qty, unit_price, order_no=ordno,
                          source="MANUAL_SYNC", order_type="MANUAL",
                          (SELL이면) profit_rate, profit_amount })
                        → 로컬 JSON + 구글시트 + 텔레그램 알림 (기존 함수가 다 함)
```

#### 핵심 함수 시그니처

```python
def get_today_executions(code: str = "") -> list:

    오류 시 빈 리스트. (chegb='1' 체결만, medosu='0' 전체)
    페이지네이션(cts_ordno)·호출제한 백오프 포함."""
```

**`trade_ledger.py` (추가)**
```python
def order_already_recorded(stock_code: str, order_no: str) -> bool:
    """trade_ledger.json에 같은 종목코드+주문번호가 이미 있으면 True.
    order_no가 빈 값이면 항상 False(판단 불가 → 기록 허용 안 함은 호출부에서 결정)."""
```

**`balance_sync.py` (추가)**
```python
def _record_manual_executions(code, name, avg_buy_price=0):
    """해당 종목의 당일 체결을 조회해, 원장에 없는 것만 MANUAL_SYNC로 기록.
    SELL이고 avg_buy_price>0이면 수익률·수익금 계산해 첨부."""
```

  원장에 없는 것만 MANUAL_SYNC 기록(매도면 수익률·수익금 계산). 예외는 안에서 삼켜 동기화 중단 방지.
- `run_balance_sync()`의 3개 분기(①삭제 ②편입 ③수량수정)에서 대상 종목을 `to_record`에 모았다가,
  파일 저장 후 한꺼번에 `_record_manual_executions` 호출.
  - case ①은 기록 삭제 **직전**에 종목명·평균매입가를 확보(수동 매도 수익 계산용).

## 📝 메모 및 특이사항

- 이 작업은 **조회(읽기) + 원장 기록**만 한다. 새로운 주문(매수/매도)은 절대 내지 않는다.

  이 계획은 거기에 "체결 기록 남기기"를 얹는 형태가 될 가능성이 높다 (P1에서 확인).
