## 배경 분석

### 문제 상황

자동매매 도중 수동으로 일부 종목을 팔거나 사면,
`held_stock_record.json`(시스템이 기억하는 보유 현황)과
실제 LS증권 계좌 잔고가 어긋나게 된다.

이 상태로 자동매매가 계속 돌면:
- 이미 판 종목에 손절 주문을 또 넣으려 할 수 있음
- 피라미딩 수량 계산이 틀어짐
- 감시해야 할 종목을 빠뜨림

### 동기화 기준

**실제 잔고를 기준(진실)으로** `held_stock_record.json`을 맞춘다.

| 상황 | 처리 방법 |
|------|-----------|
| 기록엔 있는데 실제로 없음 | 기록에서 삭제 + 텔레그램 알림 |
| 실제로 있는데 기록에 없음 | 알림만 발송 (터틀 파라미터 불명 — 자동 추가 안 함) |
| 둘 다 있는데 수량이 다름 | 실제 수량으로 기록 업데이트 + 텔레그램 알림 |
| 일치 | 아무것도 안 함 |

### "실제로 있는데 기록에 없음" — 자동 추가를 안 하는 이유

터틀 전략은 `stop_loss_price`(손절가), `next_pyramid_price`(피라미딩 기준가),
`current_unit` 등의 파라미터가 있어야 정상 작동한다.
수동 매수 종목은 이 값들을 알 수 없으므로 자동으로 집어넣으면 오히려 더 위험하다.
**알림만 보내고 사용자가 직접 판단하게 한다.**

## PHASE 3 — 구현

### P3-1 `balance_sync.py` 신규 파일 작성

**역할:** 실제 잔고 ↔ `held_stock_record.json` 비교·동기화

**함수 1개:**

```python
def run_balance_sync() -> bool:
    """실제 잔고와 held_stock_record.json을 비교해서 불일치를 수정한다.

    Returns:
        True  — 동기화 성공 (불일치가 없거나 수정 완료)
        False — 잔고 조회 실패 (ls_client.get_balance()가 빈 리스트 반환)
    """
```

**처리 흐름:**

```
1. ls_client.get_balance() 호출
   → 빈 리스트면 False 반환 (API 오류 상황 — 실행 중단 신호)

2. held_stock_record.json 읽기
   → 파일 없으면 빈 dict로 처리

3. 비교 및 수정
   ① 기록에 있는데 실제로 없는 종목
      → held_stock_record에서 해당 종목 삭제
      → 텔레그램: "⚠️ [잔고동기화] {종목명}({코드}) — 실제 잔고 없음. 기록 삭제함."

   ② 실제로 있는데 기록에 없는 종목 (감시 목록 여부 무관)
      → held_stock_record 변경 없음
      → 텔레그램: "⚠️ [잔고동기화] {종목명}({코드}) {qty}주 — 기록 없는 보유 종목 발견. 수동 확인 필요."

   ③ 둘 다 있는데 수량이 다른 종목
      → held_stock_record의 total_qty를 실제 수량으로 교체
      → 텔레그램: "⚠️ [잔고동기화] {종목명}({코드}) 수량 불일치: 기록 {기록수량}주 → 실제 {실제수량}주로 수정."

4. 변경사항이 1건 이상이면 held_stock_record.json 파일에 저장

5. True 반환
```
